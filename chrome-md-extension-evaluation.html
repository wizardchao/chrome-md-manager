⚠️ 二、核心技术挑战 — 浏览器文件系统访问

Chrome 扩展访问本地文件系统是该项目的最大技术挑战。浏览器出于安全沙箱限制，不能直接随意读写本地文件。以下是可选方案对比：

✅ 方案 A：File System Access API（推荐）

  • 原理：W3C 标准 API，Chrome 86+ 支持
  • 能力：读取、写入、删除文件与目录
  • 权限：用户主动选择目录后授权，可持久化权限
  • 优点：无需额外安装，纯前端实现
  • 缺点：首次需用户手动选择目录；仅限 Chromium 内核浏览器
  • API：showDirectoryPicker()、FileSystemDirectoryHandle

🔧 方案 B：Native Messaging + 本地服务

  • 原理：Chrome 扩展通过 Native Messaging 调用本地程序
  • 能力：完整的文件系统访问权限
  • 权限：需用户安装 Native Host 程序
  • 优点：无文件系统限制，可后台监听变更
  • 缺点：需额外安装本地程序，增加部署复杂度
  • 适用：需要监听文件变更、深度集成的高级场景

方案选择建议

维度 File System Access API Native Messaging
安装复杂度 低 仅装插件 高 插件 + 本地程序
文件读写 ✅ 完整支持 ✅ 完整支持
文件删除 ✅ removeEntry() ✅ 原生 fs 操作
目录遍历 ✅ 异步迭代器 ✅ 原生 fs 操作
文件监听 ❌ 不支持 ✅ fs.watch
跨平台分发 简单 需各平台打包
开发成本 低 中高

结论：推荐方案 A（File System Access API），满足全部核心需求，零额外安装成本，开发周期短。

5.2 文件系统核心操作

// 打开目录并遍历 Markdown 文件 async function openDirectory() { const dirHandle = await window.showDirectoryPicker(); const files = []; await scanDirectory(dirHandle, '', files); return { dirHandle, files }; } async function scanDirectory(dirHandle, path, results) { for await (const [name, handle] of dirHandle.entries()) { const fullPath = path ? `${path}/${name}` : name; if (handle.kind === 'directory') { await scanDirectory(handle, fullPath, results); } else if (name.endsWith('.md')) { results.push({ name, path: fullPath, handle }); } } } // 读取文件内容 async function readFile(fileHandle) { const file = await fileHandle.getFile(); return await file.text(); } // 写入文件内容 async function writeFile(fileHandle, content) { const writable = await fileHandle.createWritable(); await writable.write(content); await writable.close(); } // 新增文件 async function createFile(dirHandle, fileName) { const fileHandle = await dirHandle.getFileHandle( fileName, { create: true } ); return fileHandle; } // 删除文件 async function deleteFile(dirHandle, fileName) { await dirHandle.removeEntry(fileName); }

🛡️ 八、风险与应对策略

高 File System Access API 兼容性

风险：该 API 仅 Chromium 内核浏览器支持（Chrome、Edge），Firefox/Safari 不支持。
应对：明确定位为 Chrome 扩展，在说明中标注浏览器要求；降级方案使用 <input type="file"> 支持单文件打开。

中 权限持久化可能失效

风险：浏览器关闭后 File System Handle 权限可能被回收，用户需重新授权。
应对：使用 IndexedDB 持久化 Handle，启动时调用 queryPermission() 检查，必要时调用 requestPermission() 重新授权，交互成本仅一次点击。

中 大文件/大目录性能

风险：包含数千文件的目录遍历可能卡顿。
应对：懒加载文件树（仅展开时遍历子目录）、Web Worker 处理 Markdown 解析、虚拟列表渲染。

低 Chrome Web Store 审核

风险：插件可能因权限描述不清被拒。
应对：详细说明权限用途，提供隐私政策，录制功能演示视频。

低 Manifest V3 限制

风险：MV3 中 Service Worker 生命周期有限，不能长驻后台。
应对：文件操作在前端页面（Side Panel / New Tab）中执行，不依赖 Service Worker 保持状态。

✅ 十、总结与建议

可行性结论：完全可行

  • 技术上 — File System Access API 提供了完整的本地文件读写能力，配合 Manifest V3 可实现全部需求，无需后端服务器。
  • 成本上 — 纯前端项目，依赖全部开源免费，独立开发者仅需投入 2-3 周时间 + $5 注册费。
  • 体验上 — 浏览器内直接使用，零安装成本（对用户而言），填补了「浏览器内本地 Markdown 文件管理」的市场空白。

推荐实施路径

  1. MVP（最小可行产品）：先实现打开目录 + 查看 + 基础编辑，约 1 周可完成
  2. V1.0：加入完整编辑器、文件管理、UI 美化，约 2-3 周完成
  3. V2.0：考虑增加 Git 集成、多标签页编辑、插件主题系统等高级功能
